Agent EvaluationKernel Optimization

An agent evaluation targeted at optimization and graded with an ELO mechanism.

S / T

(1) Current agent evaluation focuses on correctness, but in practice many problems are optimization problems with no absolute right or wrong. (2) Unlike optimizers such as AdamW, an agent tuning parameters often abruptly increases or decreases a parameter when it sees no obvious improvement, causing it to miss the chance of reaching the optimum. Optimization problems therefore test whether an agent can align well with human instructions in a specific optimization scenario.

A

Built three scenarios (Game: a modified Halite III; AI Infra: kernel optimization on H100; Finance: stock trading), containerized with Docker for sandbox isolation to make later agent testing convenient. The first scenario is complete; the remaining work is still in progress.

2026.02 — 2026.03

Multi-Role RL — Planner / Coder / Reviewer

Post-TrainingAgentic MARLReinforcement Learning

Reproduction of arXiv:2602.01244 plus Agentic MARL. (2026.02 – 2026.03)

S / T

Coding agents in the CLI face inherent limitations: planning and execution are decoupled, credit assignment is ambiguous, and role capabilities are highly entangled. Built on the Qwen3-Coder-30B-A3B-Instruct base model, this project uses TerminalTraj (50k single-agent command sequences) and its Docker environment to build a Planner–Coder–Reviewer three-role collaborative multi-agent system, aiming to break the single-agent performance ceiling on TerminalBench (1.0 / 2.0).

A

Before training, the 50k single-agent shell trajectories in TerminalTraj were functionally decoupled and cleaned, and new data was mined using the paper's method to adapt them to the Coder role's SFT and RL environment. Training used a staged frozen strategy: the Coder extracts successful trajectories for SFT, then runs GRPO in Docker rewarded by command exit codes and task-completion flags; the Planner reverse-distills successful trajectories into "high-level sub-goal sequences" and is fine-tuned with search-based GRPO using the Coder's validation-set command pass-rate as the reward signal for policy alignment; the Reviewer pairs "failed command + environment error" with "corrected successful command" to generate diagnostic review notes, and is trained with SFT only for stability. The three share one base model, form a collaborative pipeline at inference, and are measured with fine-grained metrics such as plan-executability, first-round pass-rate, and repair-rate.

R

The approach achieved a significant breakthrough on TerminalBench. TB 2.0 (overall goal): baseline + React (9.6%) → Coder-only + React (17.4%) → MultiModel + Plan-and-Execute (23.2%). TB 1.0 (Coder only): baseline (15.3%) → Coder-only (33.0%) → MultiModel (28.3%). This shows that even though the Coder's own capability drops under the Planner/Reviewer roles, the agent as a whole becomes stronger — demonstrating the feasibility of Agentic MARL.

2026.04 — 2026.05

vLLM Inference & Kernel Optimization on RTX 4050 & A800×2

Kernel OptimizationInference OptimizationvLLM

Four experiments across a RTX 4050 laptop and A800×2. (2026.04 – 2026.05)

Experiment ① — RTX 4050 laptop · Qwen3.5-0.8B · kernel optimization
S / T

FlashInfer PagedAttention decode underperforms on small GPUs such as the RTX 4050 laptop under small-batch, medium-to-long-sequence scenarios.

A

Traced the scheduler.cuh, decode kernel, and JIT dispatch call chain, and built a batch-size / sequence-length benchmark matrix. Then introduced fixed_split_size and disable_split_kv tuning knobs, and adjusted the split heuristic for small-GPU workloads to reduce ineffective partition and merge.

R

In typical scenarios, B1/S4096, B2/S4096, and B8/S4096 latency dropped by about 12%, 11%, and 14% respectively, with effective throughput rising in step; the gains come mainly from lower scheduling overhead.

Experiment ② — RTX 4050 laptop · Qwen3.5-0.8B · end-to-end inference optimization
S / T

Integrate the local FlashInfer into vLLM to verify how kernel-level gains affect real LLM serving, and further optimize TTFT, latency, and token/s.

A

Built a vLLM offline benchmark and systematically swept configurations. Found that Qwen3.5-0.8B relies on full attention in only some layers, and the end-to-end bottleneck comes more from GDN / linear attention and vLLM graph / scheduling — so the optimization focus was extended from PagedAttention to the serving-level execution path.

R

Versus the eager baseline, B1 latency dropped from ~2.84s to ~0.68s, and output throughput rose from ~22.5 token/s to ~94 token/s; B16 aggregate throughput reached ~999 token/s, clarifying the benefit boundary of PagedAttention optimization on the end-to-end path.

Experiment ③ — A800×2 · Qwen3-30B-A3B · kernel optimization
S / T

From several candidate operator paths, select the truly production-ready combinations while keeping experimental evidence — making clear which schemes are only suitable for diagnosis and which should be rejected. Key targets include fused SiLU + W2 pointer setup, W2 direct-warp routed-sum, bf16x2, CUDA topk, W1 direct/CUTLASS, route-fold, WMMA, moe_sum, align hybrid, and the FlashInfer sampler new API.

A

Optimized the analyzed directions through three tiers of validation: microbench, integrated fused_experts, and online A/B.

R

In testing, every item gained about 5%–15% performance improvement within that kernel's scope.

Experiment ④ — A800×2 · Qwen3-30B-A3B · end-to-end inference optimization
S / T

Integrate the above kernel optimizations into vLLM to observe whether all of them help inference. The service runs in a vLLM environment with the FlashInfer attention backend, mainly observing c32 and c64.

A

Systematically organized the benchmark configs and designed / validated multiple CUDA/CUTLASS paths around W2. Finally kept fused SiLU + W2 pointer setup (removing one CUDA launch); introduced W2 direct-warp routed-sum (skipping pair-output and extra summation); and combined bf16x2 with K=384 unroll to extend direct-warp applicability from M≤2 to M≤8, while filtering out the remaining unstable schemes.

R

In the short 128/128 c32 online test, versus the max2 baseline, output throughput rose ~20.2%, p99 TTFT dropped ~72.8%, p99 TPOT dropped ~14.1%, and p99 E2E dropped ~28.3%.

2026.05 — 2026.06

MoE Expert-Parallel Communication Optimization

Expert ParallelInference OptimizationProfiling

Expert-parallel inference profiling around SGLang + DeepEP on 4×A100.

S / T

In a 4×A100 environment, set up expert-parallel inference experiments around SGLang + DeepEP to analyze how communication overhead affects time-to-first-token, per-token latency, and output throughput under different concurrency, input lengths, and expert-routing loads.

A

Chose Qwen3-30B-A3B as the test target and, fixing the model, batch, and input/output lengths, tested default communication, DeepEP normal mode, and low-latency mode separately. Recorded total request time, prefill time, decode time, and per-layer expert dispatch/combine time, with a script auto-generating comparison tables across concurrency levels. To locate bottlenecks, used NSYS to capture the timeline across the 4 A100s — focusing on communication kernels, expert-compute kernels, and scheduling gaps — then used NCU to inspect the bandwidth utilization and launch overhead of key communication kernels. Based on results, switched low-concurrency scenarios to low-latency mode and high-concurrency scenarios to normal mode, and compiled communication strategies for three scenario classes: input length, concurrency size, and expert-load imbalance.

R

Produced a reproducible MoE distributed-inference profiling workflow on 4×A100. In low-concurrency decode scenarios TTFT dropped by ~15%–25%, and in high-concurrency scenarios output throughput rose by ~10%–18%; the share of communication time in end-to-end inference and its optimization boundary were also clarified.

2026.03 — 2026.05

Full-Stack Q&A Generation Agent for Data Synthesis

AgentLangGraphData Synthesis

A data-synthesis Q/A generation agent from a paper project. (2026.03 – 2026.05)

S / T

Today's LLMs are weak at creatively generating logical-reasoning problems (especially puzzles), mainly because they are weak at understanding puzzle rules and enumerating every case; they badly need an external harness to correct them before they can generate high-quality Q/A pairs for later mid-training and post-training.

A

Built a Generator–Solver–Reviewer multi-agent pipeline on LangGraph, integrating rule parsing, structural validation, cross-solving, and deduplication; added a user-behavior memory module that records the user's preferred rules, problem types, difficulty, feedback, and failure modes for cross-session personalized generation and proactive recommendation; and built an Agent Benchmark with fixed rules, seeds, and metrics covering Anagram, 24, Sudoku, and Skyscrapers, implementing a Self-Evolution loop: automatically generating prompt/strategy candidates from failure attribution and switching the active prompt version after a regression gate.

R

The project produced a reproducible puzzle Q/A generation, evaluation, and self-evolution system; semantic-probe tests passed 10/10, and metrics such as acceptance rate, structural pass-rate, cross-validation pass-rate, duplication rate, and latency quantify each agent version's gains — fully demonstrating agent memory, self-correction, self-evolution, and engineering delivery.

2025.10 — 2025.12

AGRO-SQL — Agentic RL for Text-to-SQL

Agentic RLText-to-SQLGRPO

A practice of agentic RL for Text-to-SQL. (2025.10 – 2025.12)

S / T

Progress in Text-to-SQL has long been constrained by two bottlenecks: a severe shortage of high-quality training data, and insufficient model reasoning on complex queries. Samples from traditional data-synthesis methods suffer from logical misalignment and semantic deviation, hard to use directly for RL training. A holistic solution is needed to crack both "data scarcity" and "weak model reasoning" and push Text-to-SQL to a more advanced level.

A

Designed a two-stage agentic RL framework: stage one uses a "diversity-aware cold start" to build a robust initial policy from high-quality trajectories; stage two introduces GRPO (Group Relative Policy Optimization), replacing traditional value-network methods by using the relative execution rewards of a group of synthesized trajectories as the optimization basis, iteratively refining the policy through group competition and effectively solving exploration efficiency under sparse rewards.

R

On the two authoritative benchmarks Spider and BIRD, AGRO-SQL showed top-tier single-model performance — based on Qwen3-8B-Base, it achieved 72.10% execution accuracy on BIRD.

2026.06 — 2026.07

Efficient AWQ Quantization Tuning (LLM-Compressor)

QuantizationAWQInference Optimization

Model-quantization tuning: efficient AWQ based on LLM-Compressor.

S / T

When deploying Qwen3-32B, it needed 4-bit weight quantization (W4A16) to reduce memory usage. Initially using llm-compressor v0.6.0 with default settings for AWQ, the model scored significantly below the industry benchmark AutoAWQ on GSM8K and MMLU, and generated text with severe formatting chaos (many invalid "*" symbols). This project locates the root cause of llm-compressor's quantization degradation.

A

Comparing llm-compressor and AutoAWQ on hyperparameters (calibration sample count, max sequence length) and core algorithm, and ruling out sample-count effects, pinpointed the key difference as the quantization observer type. Source and community analysis revealed llm-compressor defaults to a minmax observer, whereas AutoAWQ actually uses a clipping-based MSE error-minimization strategy. Modified the AWQModifier config to explicitly set observer="mse", kept 128 calibration samples and sequence length 4096, and re-ran quantization.

R

With the mse observer, GSM8K accuracy rose from 0.848 to 0.884, close to AutoAWQ; text formatting returned to normal and dialogue quality clearly improved.

On the page

EMNLP · Under Review · Co-First Author

Everything in Moderation: Dose-Response Effects of Data Coverage and Alignment-Resistant Capability Gaps in Multi-Domain Mid-Training

Yunpeng Xu (co-first author) · 2026

A systematic test of an unexamined assumption — that uneven domain coverage during mid-training can simply be corrected by later SFT/RL. Across nine coverage configurations on Qwen3-8B-Base, I find that (1) domain accuracy follows an inverted-U dose-response curve with an identifiable "moderate" regime, and (2) capability gaps seeded in mid-training stubbornly persist into alignment — SFT actually widens them, and compensatory SFT can't reliably close them under a fixed budget.
